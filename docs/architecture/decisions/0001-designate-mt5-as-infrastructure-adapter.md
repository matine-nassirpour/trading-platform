# ADR 0001 — Placement of the MT5 Macro-Adapter

> **Status:** Accepted
> **Date:** 2025-10-28
> **Version:** 1.0.0
> **Compliance Level:** Level 3 — Critical (Immutable Architectural Invariant)

---

## 1. Context

The `mt5` subsystem is a composite outbound adapter responsible for translating
execution intents generated by the Python Core / Application Layer into
concrete MetaTrader 5 API operations. It manages execution resilience,
broker-specific policies, observability, and cross-broker consistency.

The component is multi-module, combining session management, runtime state,
health monitoring, and request/response mapping.

---

## 2. Decision

Place the MT5 subsystem under:
`src/quantum/infrastructure/adapters/mt5/`

This location classifies MT5 as an **Outbound Infrastructure Adapter** according
to the Clean Architecture pattern.

---

## 3. Rationale

- **Architectural Principle:**  MT5 translates internal domain intentions into external API calls.
It implements outbound ports from the Application Layer.
- **Clean Architecture Alignment:** Infrastructure layer hosts all technical adapters and gateways
to external systems. The MT5 subsystem fulfills that definition.
- **Dependency Inversion Principle:** The Application Layer defines abstract ports.
MT5 provides their concrete implementations without creating upward dependencies.
- **Safety-Critical Conformance:**
  - *DO-178C §5.3.3:* MT5 is a "Software Interface Component" — isolated and traceable.
  - *ISO 26262-6 §7.4.7:* MT5 is an external interface, separated from application logic.
  - *NASA-STD-8739.8 §5.5.3:* External interfaces must be independently verifiable.
  - *IEC 62304 §5.3.2:* Infrastructure software isolated from application software.
- **Auditability:**  Placement under `infrastructure/adapters/` ensures alignment with the Import Linter
governance rules (Contract 1: Clean Architecture Layering).

---

## 4. Consequences

✅ The MT5 subsystem is verifiable, mockable, and isolated from business logic.
✅ Cross-broker adapters (FTMO, FundedNext, etc.) can coexist under the same structure.
✅ Observability and runtime elements remain within Infrastructure scope.
❌ Moving it under `application/` or `platform/` would break layer isolation.

---

## 5. Alternatives Considered

| Option                                  | Reason for Rejection                                                             |
|-----------------------------------------|----------------------------------------------------------------------------------|
| `src/quantum/application/adapters/mt5/` | Violates the dependency direction (Application cannot depend on Infrastructure). |
| `src/quantum/platform/mt5/`             | Platform is for internal runtime only, not external integrations.                |
| `src/quantum/infrastructure/mt5/`       | Semantically vague, loses the “adapter” role clarity.                            |
| `src/quantum/subsystems/mt5/`           | Introduces a non-standard layer; breaks governance rules.                        |
